目标：复盘这两天从接需求、实现、调试 bug 到生产验证的过程，沉淀成可复用的 AI 协作开发流程，并把 UI 质量纳入强制验收。

## 1. 结论

这两天 bug 多，不是因为某个单点技术特别难，而是因为开发流程缺少固定关卡：

1. PRD 理解没有变成可验收的需求锁定文档。
2. 技术方案没有提前覆盖模型、缓存、D1 状态、生产环境、并发会话和回滚。
3. 测试用例是事后补，不是实现前的准入门槛。
4. UI 质量靠主观反馈，没有在开发前定义视觉方向、组件标准和截图验收。
5. 产品人体验验收发生太晚，导致“看起来能跑”之后才发现信息层级、文案、专业度和交互问题。

## 2. 为什么会有这么多 bug

| 问题类型 | 本次表现 | 根因 | 以后怎么防 |
| --- | --- | --- | --- |
| 需求漂移 | 从 H5 Demo 到真实图片识别、报告专业表格、餐盘知识库、我的档案、硬件浮窗、行动重写不断叠加 | 没有每轮改动前写“目标 / 非目标 / 验收标准” | 每个功能先写 PRD Understanding |
| UI 反复被否 | 首页、行动、报告、配料表、硬件浮窗多次被说丑或不专业 | “好看”没有拆成可执行的视觉标准和截图验收 | 每个 UI 改动先写 UI Acceptance |
| AI 结果弱 | 报告内容少、建议泛、标准对比缺失、餐盘/配料表结构不足 | 只看模型输出，没有先定义前端结构和规则层保底 | 先定结果结构，再选模型 |
| 模型慢 | Kimi/GPT/Gemini 多轮替换，页面长时间等待 | 技术方案未提前基准测试，前端超时窗口与真实链路不匹配 | 模型接入必须先做延迟/质量基准 |
| 状态串台 | AI tab 切换后残留上一工具状态 | 前端状态模型没有作为技术风险提前评审 | 技术方案必须画状态分槽 |
| 生产与本地不一致 | 静态资源版本回退、生产缓存导致旧 UI | 部署验证没有成为每次改动的必经关卡 | 每次前端改动必须校验资源版本和生产 URL |
| 并发会话冲突 | 行动页/打卡/首页被其他会话同时修改 | 没有统一工作树策略和最新主线检查 | 广泛改动必须 worktree + fetch/rebase |
| 测试覆盖薄 | `node --check` 通过但移动端排版、产品体验仍有问题 | 测试用例偏接口和语法，没有覆盖视觉、产品、AI 输出 | 测试矩阵必须含 UI/产品/模型/生产 |
| 产品验收缺失 | 产品人截图反馈后才发现排版、文案、专业度问题 | 开发完成定义只看工程验证，不看产品验收 | Done 必须包含 Product Acceptance Report |

## 3. 建议的 Skill 工作流

主 skill：`product-development-workflow`

它不替代具体开发技能，而是作为总控流程，按阶段调用其他能力。

| 阶段 | 目标 | 必须产物 | 通过标准 |
| --- | --- | --- | --- |
| 1. 需求理解 | 把聊天、PRD、图片、旧站、用户反馈统一成明确目标 | `*_PRD_UNDERSTANDING.md` | 目标、非目标、验收标准、假设清楚 |
| 2. UI/产品设计 | 把“丑 / 不专业 / 不好用”转成视觉与体验标准 | `*_UI_ACCEPTANCE.md` 或 UI 决策记录 | 首屏任务、视觉方向、状态、截图清单明确 |
| 3. 技术方案 | 提前识别数据、状态、模型、缓存、部署风险 | `*_TECHNICAL_PLAN.md` | 数据流、接口、失败模式、日志、回滚清楚 |
| 4. 测试计划 | 先定义怎么证明它做对了 | `*_TEST_CASES.md` | 功能、UI、模型、生产、回归用例都有证据 |
| 5. 实现 | 小步开发，不把文档和代码分离 | 代码 + 文档更新 | 每次行为变化都有记录 |
| 6. 工程验证 | 证明代码和环境正确 | 命令输出、截图、API 结果、生产 smoke | 本地和生产证据都匹配验收标准 |
| 7. 产品验收 | 产品人按用户视角走完整流程 | `*_PRODUCT_ACCEPTANCE_REPORT.md` | P0 问题清零，P1/P2 入 backlog |
| 8. 复盘 | 把 bug 变成流程改进 | `docs/开发进度.md` 或复盘文档 | 新 bug 类型进入 skill 或测试模板 |

## 5. UI 为什么一直显得丑

这次 UI 的核心问题不是“缺少装饰”，而是没有稳定设计系统和验收规则：

1. 视觉方向多次摇摆：原站专业内容品牌、深色监控卡、小清新健康卡、硬件炫酷浮窗都出现过。
2. 页面目标不够聚焦：首页既想显示状态分、曲线、提醒、拍照、压力精力、最近记录，又想第一眼讲清主行动。
3. 组件密度不统一：有些地方像内容站，有些像健康 App，有些像后台表格。
4. 图表和表格补得晚：报告、餐盘、配料表一开始偏文字摘要，后来才加标准对比和红绿灯。
5. 没有每次改动后的视觉截图验收：很多问题靠用户截图反馈，而不是开发前定义和开发后自测。

## 6. UI 解决思路

不要继续零碎调色和补卡片。建议做一次 `UI Reset Sprint`，范围控制在核心流程，不重写业务逻辑。

### 6.1 先定产品视觉原则

GLUCOLIT 应该是：

- 可信的糖前期生活方式助手，不是泛 SaaS 仪表盘。
- 温和但不幼稚，不使用过度疗愈、装饰性植物和弱对比文字。
- 行动导向，首屏只让用户知道“今天最该做什么”。
- 专业内容可折叠，默认只展示结论、证据和下一步。

### 6.2 建立 UI 验收标准

每个页面必须通过这些问题：

| 页面 | 5 秒问题 | UI 验收重点 |
| --- | --- | --- |
| 首页 | 今天最重要的一件事是什么 | 1 个主结论 + 1 个主行动，指标下沉 |
| AI 工具 | 我该拍什么，结果是否可信 | 上传态、流式态、结果态清晰；报告/餐盘/配料表结构不同 |
| 报告结果 | 哪些指标正常，哪些要问医生 | 标准对比表、曲线表、专业建议分区 |
| 餐盘结果 | 这餐哪里影响餐后波动 | 餐盘占比、主食/蛋白/蔬菜/油脂结构 |
| 配料表结果 | 这个能不能常买 | 红绿灯、触发原因、替代选择 |
| 行动页 | 今天先做哪一个 | 主行动突出，可选行动不抢重点 |
| 我的页 | 我是不是在变好 | 复查故事优先，完整指标折叠 |
| 科普页 | 我的问题在哪里找答案 | 用户问题导航优先，研究证据折叠 |

### 6.3 执行方式

1. 先用 Stitch 或 UI 设计工具产出 2-3 个方向，不直接进代码。
2. 选一个方向后写 `GLUCOLIT_UI_ACCEPTANCE.md`。
3. 把核心组件统一：按钮、标签、卡片、表格、折叠、导航、状态条、分析结果块。
4. 每改一个页面都输出 360/390/桌面壳截图。
5. 产品人按 `Product Acceptance Report` 走完整链路，P0 不过不算完成。

### 6.4 不建议继续做的事

- 不要继续每个页面单独换颜色。
- 不要继续堆更多卡片解决专业感。
- 不要把所有专业指标默认展开。
- 不要把 AI 输出当长文展示。
- 不要只靠 hover、阴影、渐变制造“高级感”。

## 7. 下一次类似项目的执行顺序

以后新项目或大功能建议按这个节奏：

1. **半小时需求锁定**：输入源、目标、非目标、验收标准。
2. **半小时 UI/产品方向**：首屏、主路径、视觉参考、产品验收问题。
3. **半小时技术方案**：接口、数据、模型、缓存、部署、失败模式。
4. **半小时测试矩阵**：功能、UI、模型、生产 smoke。
5. **小步实现**：每次只改一个闭环。
6. **本地验证**：命令 + Playwright + API。
7. **生产验证**：资源版本、接口、真实流程。
8. **产品人验收**：截图反馈进 P0/P1/P2。
9. **复盘更新 skill**：把新 bug 类型写回流程。

## 8. 对当前 GLUCOLIT 的建议

短期不要继续扩大功能。建议优先做三件事：

1. **做 UI Reset Sprint**
   - 首页、AI 结果、行动页、我的页先统一视觉和信息层级。
   - 先产出 UI 验收文档，再改代码。

2. **把 AI 结果变成结构化产品组件**
   - 报告：标准对比 + 曲线结构 + 医生问题。
   - 餐盘：占比结构 + 重点信号 + 低门槛替换。
   - 配料表：购买红绿灯 + 触发原因 + 替代选择。

3. **补齐产品验收流程**
   - 每次上线后由产品人按固定账号、固定路径、固定截图清单验收。
   - 验收报告写入仓库，不再只靠聊天反馈。
